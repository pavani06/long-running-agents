# Symphony Trap Awareness

**Type:** canonical
**Status:** active
**Source:** analysis/2026-06-12-idsd-method/ and analysis/2026-06-11-the-trap-spec-driven-development-is-setting/
**Classification:** Partial Coverage, Medium integration value
**Precedence:** document-level 2 (canonical) per [[docs/system-of-record|System of Record]]

---

## Problem

The most impressive agentic specification ever produced -- OpenAI's Symphony, 2,169 lines, eighteen sections, formal must-and-should language -- was not written upfront. It was reverse-engineered from a system that was already running. OpenAI spent ~6 months building the internal tool under a rule of zero human code, every line generated by Codex. Only after it worked did they distill the spec from the running system. Then they had Codex rebuild it in Elixir in one shot, and separately implement the same spec in five languages (TypeScript, Go, Rust, Java, Python) to shake out ambiguities.

The industry sells the output as if it were the method. The trap: "A industria esta vendendo o output como se fosse o metodo. Nao se pode escrever a spec upfront. A unica organizacao que produziu uma spec tao boa a produziu por ultimo, fazendo engenharia reversa de software que ja estava vivo" ([[docs/analysis/2026-06-12-idsd-method/2026-06-12-idsd-method-analysis|analysis]]:49-51).

The failure pattern is not that specifications are bad. It is that treating polished retrospective specifications as upfront inputs leads teams to demand a precision that usually exists only after the system has run. A specification of that quality emerges from a system that has encountered real constraints, real users, and real failure modes -- not from speculative upfront writing.

The repo's philosophy is explicitly anti-upfront-spec. The "The Trap Spec-Driven Development Is Setting" analysis names this trap directly. Code is framed as a disposable build artifact in [[docs/canonical/llm-as-fuzzy-compiler|LLM as Fuzzy Compiler]]. However, the specific ritual of reverse-engineering specs from running systems (ambiguity probes, multi-implementation shaking, behavior distillation) is not formalized as a named practice ([[docs/analysis/2026-06-12-idsd-method/2026-06-12-idsd-method-classification|classification]]:203-216).

## Solution

Internalize that useful specifications emerge from running systems, not from upfront writing. Replace "write spec then build" with "build then distill spec from what works then rebuild with the spec as a contract." The running system is the source of truth; the spec is documentation of observed behavior.

**The real process versus the marketed process:**

| Marketed process | Real process (Symphony) |
|---|---|
| Write spec upfront | Build prototype until it works (zero human code, six months) |
| Give spec to agent | Observe what the running system actually does |
| Agent builds from spec | Extract success patterns, failure modes, and constraints from observed behavior |
| Done | Distill into a formal spec document |
| | Use spec to rebuild (Elixir one-shot, five-language ambiguity shaking) |
| | Spec now serves as regression contract for future agents |

The step the industry skips in its marketing is also the most expensive and the most uncertain: building something that works first, in the dark, without a spec.

**Running-System Spec Distillation ritual:**

When a spec-quality artifact is needed for agent reuse, derive it from a running system, not from speculation:

1. **Build a working prototype or slice.** Something real that runs, even if incomplete. The system must have encountered at least some real constraints and failure modes.
2. **Observe behavior.** Record what the system does: passing behaviors, failure cases, edge conditions, real user interactions, and discovered constraints.
3. **Ambiguity probes.** Attempt independent rebuilds or alternate implementations of the same behavior. Each rebuild that diverges reveals an ambiguity in the implied spec -- a place where two reasonable implementers would make different choices.
4. **Owner validation.** The outcome owner reviews the observed behaviors and separates desired behavior from accidental implementation. Not everything the running system does is what it should do.
5. **Distill the reference behavior set.** From desired behaviors and discovered constraints, produce a reference artifact: test cases, eval targets, rebuild prompts, or a formal spec. This artifact is grounded in observed reality, not upfront imagination.
6. **Use as a regression contract.** The distilled spec becomes a durable contract for future agent runs. Because it was extracted from a system that worked, it encodes real constraints rather than speculative ones.

**What the Symphony Trap is not saying:**

The trap is not "never write specifications." The trap is "do not mistake a retrospective artifact for a prospective method." Specifications are useful -- as regression contracts for systems that already work. They become harmful when teams demand upfront the kind of precision that only emerges from a running system. The harm is not that they fail to write the spec -- it is that they spend weeks trying to, while the agent could have been building and learning from real behavior.

## Implementation in this repo

### What already exists

- [[docs/analysis/2026-06-11-the-trap-spec-driven-development-is-setting/2026-06-11-the-trap-spec-driven-development-is-setting-analysis|The Trap SDD Analysis]]:1-280 is a complete analysis of the upfront-spec trap. Its classification labels the repo's Intent-First Spec Loop as Better Implementation -- the repo's intent-first pipeline exceeds upfront spec approaches.
- [[docs/canonical/llm-as-fuzzy-compiler|LLM as Fuzzy Compiler]]:27-28 frames code as "a disposable build artifact" and states that "what matters is preserving the prompts, guardrails, and documentation that produced the code, not the code itself." This is the repo's strongest anti-upfront-spec statement: the durable asset is the harness, not the spec.
- [[docs/canonical/production-grounded-eval-sampling|Production-Grounded Eval Sampling]]:26-52 anchors evaluation in representative production interactions, traces, tool results, and state snapshots. Grounds validation in real behavior rather than upfront expectations -- implementing the principle that ground truth comes from running systems.
- [[docs/canonical/production-failure-regression-flywheel|Production Failure Regression Flywheel]]:28-60 converts production failures into durable regression knowledge. Failures observed in running systems become the spec for future behavior.
- [[docs/canonical/eval-to-production-correlation-tracking|Eval-to-Production Correlation Tracking]]:28-50 tracks correlation between eval scores and production outcomes. If eval scores diverge from production behavior, the evals -- not the running system -- are updated.
- [[docs/canonical/constraint-anchored-evaluation|Constraint-Anchored Evaluation]]:29-56 anchors evaluation on constraints from client state and business rules that exist in the running system.

### What is missing from the pattern

The classification marks Symphony Trap Awareness as Partial Coverage because the repo embodies the philosophy (anti-upfront-spec, production-grounded, code as disposable) but does not formalize the ritual of reverse-engineering specs from running systems as a named practice ([[docs/analysis/2026-06-12-idsd-method/2026-06-12-idsd-method-classification|classification]]:203-216).

Missing items:

1. The Symphony Trap as a named pattern with the explicit warning: do not mistake retrospective spec quality for prospective spec method.
2. The Running-System Spec Distillation ritual: build prototype -> observe behavior -> ambiguity probes -> owner validation -> distill reference behavior set -> use as regression contract.
3. Ambiguity probes as a named practice: independent rebuilds or alternate implementations to shake out hidden ambiguities in the implied spec.
4. The distinction between desired behavior and accidental implementation: not everything the running system does is what it should do.
5. Formal connection between the Symphony Trap awareness and the existing production-grounded eval infrastructure: the eval pipeline already grounds validation in running systems; the distillation ritual formalizes how to produce the reference behavior artifacts.

## Tradeoffs

| Benefit | Cost |
|---|---|
| Grounds specs in software that has already encountered real constraints, users, and failure modes | Requires something real enough to observe -- cannot replace minimal upfront intent and safety constraints |
| Avoids the Symphony Trap of mistaking a final artifact for the method that produced it | Multi-implementation ambiguity shaking is expensive in time and tokens |
| Creates stronger regression material for future agent runs than speculative upfront prose | Retrospective specs can encode accidental behavior unless owners separate desired behavior from incidental implementation |
| Complements the repo's existing production-grounded eval infrastructure with a named ritual for producing reference artifacts | The market pressure to produce "beautiful specs" upfront is strong and the distinction (output vs. method) is subtle |
| The ambiguity probes reveal gaps that no amount of upfront writing would have caught | Not suitable for greenfield work where no running system exists; in those cases, the ICE crafts + iterative build still apply |

- **[[docs/canonical/production-grounded-eval-sampling|Production-Grounded Eval Sampling]]** -- the eval sampling already grounds validation in running systems. The Symphony Trap awareness explains why: upfront expectations are speculative; running-system behavior is real.
- **[[docs/canonical/production-failure-regression-flywheel|Production Failure Regression Flywheel]]** -- the flywheel converts failures from running systems into durable regression knowledge. This is the same principle applied to failure: the spec for what must not happen comes from what did happen.
- **[[docs/canonical/eval-to-production-correlation-tracking|Eval-to-Production Correlation Tracking]]** -- when eval scores diverge from production behavior, the evals -- not the running system -- are updated. This is Symphony Trap awareness applied to evaluation: the running system is the source of truth; evals are documentation.
- **[[docs/canonical/generator-evaluator|Generator-Evaluator]]** -- the Evaluator validates against expectations. The Symphony Trap warns that expectations written upfront may encode assumptions the running system would contradict. Production-grounded expectations are more reliable.
- **[[docs/canonical/ice-craft-separation|ICE Craft Separation]]** -- ICE separates intent from expectations from context. The Symphony Trap extends ICE by saying that high-quality Intent and Expectations artifacts often emerge from running systems rather than upfront writing. ICE defines the structure; Symphony Trap defines the method for populating that structure with real data.
- **[[docs/canonical/constraint-anchored-evaluation|Constraint-Anchored Evaluation]]** -- constraints from client state and business rules come from the running system. The Symphony Trap ensures we do not replace those real constraints with speculative ones written upfront.
- **[[docs/canonical/application-owned-agent-control-plane|Application-Owned Agent Control Plane]]** -- the control plane orchestrates agent execution. The Symphony Trap ensures the control plane does not demand upfront precision that can only come from a running system.
- **[[docs/canonical/value-gated-agent-control-loop|Value-Gated Agent Control Loop]]** -- the value gate asks "is this worth building?" The Symphony Trap adds: "and if we define the spec upfront, are we defining something that a running system would validate or contradict?"
