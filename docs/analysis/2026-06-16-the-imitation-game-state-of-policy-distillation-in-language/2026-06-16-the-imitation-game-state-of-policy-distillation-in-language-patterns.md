# Reusable Agentic Patterns from The Imitation Game

Scope: extracted from [[docs/analysis/2026-06-16-the-imitation-game-state-of-policy-distillation-in-language/2026-06-16-the-imitation-game-state-of-policy-distillation-in-language-analysis|Policy Distillation Analysis]] and oriented by the Phase 0 repository mental model. Only mechanisms with a clear mapping to long-running agent workflows were kept. Weak analogies called out by Phase 1, such as tokenizer mismatch as agent action-space mismatch and rock tokens as a standalone agent pattern, were not promoted into patterns.

## 1. On-Policy Rollout Feedback Loop

Provenance: Phase 1 sections 1.1 and 6 (`2026-06-16-the-imitation-game-state-of-policy-distillation-in-language-analysis.md:33`, `2026-06-16-the-imitation-game-state-of-policy-distillation-in-language-analysis.md:235`); source references l44-53 and l122-144.

- **name:** On-Policy Rollout Feedback Loop
- **problem solved:** Agents trained or evaluated only on curated scripts fail when their own production trajectories contain early mistakes, strange tool outputs, or context drift that never appeared in the static data.
- **inputs:**
  - Production-like prompts, replay prompts, or live task seeds.
  - Current agent loop, prompt, tools, context builder, and state policy.
  - Full trajectory trace with model decisions, tool calls, tool results, errors, and context snapshots.
  - Teacher, evaluator, verifier, or human review signal over the agent's own prefixes.
  - Update target such as prompt rules, skills, eval cases, memory policy, or model fine-tuning data.
- **outputs:**
  - On-policy trajectory dataset generated by the current agent rather than by an idealized teacher.
  - Step and trajectory feedback attached to the actual prefixes the agent produced.
  - Updated harness, prompts, evals, or training examples that reflect the inference distribution.
- **benefits:**
  - Closes the exposure-bias gap between clean training examples and messy runtime trajectories.
  - Turns production drift, failed tool calls, and late-session context errors into high-value learning data.
  - Makes compounding errors visible before they become silent O(eT^2)-style cascades in long workflows.
- **limitations:**
  - Rollouts are expensive and can be low quality early in the agent lifecycle.
  - Requires privacy, safety, and sampling controls before production traces become learning data.
  - Needs a reliable evaluator; otherwise the loop converts runtime noise into training noise.

## 2. Autonomy Curriculum Sampling

Provenance: Phase 1 section 2.1 (`2026-06-16-the-imitation-game-state-of-policy-distillation-in-language-analysis.md:90`); source references l177-178, l187-205, and l224.

- **name:** Autonomy Curriculum Sampling
- **problem solved:** Fully supervised agents never practice autonomous recovery, while fully autonomous agents produce unusable traces before their loop is stable.
- **inputs:**
  - Expert, human, or stronger-agent demonstrations.
  - Current-agent rollouts on the same task family.
  - Autonomy mix parameter such as `lambda`, readiness thresholds, or phase gates.
  - Metrics for task success, evaluator confidence, repair rate, and unsafe-action rate.
  - Schedule policy that decides when to increase self-generated work.
- **outputs:**
  - Mixed curriculum that starts teacher-heavy and gradually increases self-generated trajectories.
  - Explicit autonomy schedule for each task class or agent capability.
  - Gate evidence showing when the agent can move from observe to assist to own.
- **benefits:**
  - Reduces cold-start collapse by avoiding pure student rollouts before the agent can recover from basic errors.
  - Creates one visible dial for autonomy instead of an implicit jump from manual operation to full agent ownership.
  - Keeps the agent exposed to its own mistakes as soon as those mistakes become learnable.
- **limitations:**
  - Requires tuning the schedule; too slow hides autonomy failures, too fast floods the loop with noise.
  - Demonstrations can overfit the agent to expert paths and reduce exploration.
  - Readiness metrics must be task-specific enough to avoid promoting brittle autonomy.

## 3. Privileged Context Self-Distillation

Provenance: Phase 1 sections 1.3, 2.2, and 6 (`2026-06-16-the-imitation-game-state-of-policy-distillation-in-language-analysis.md:60`, `2026-06-16-the-imitation-game-state-of-policy-distillation-in-language-analysis.md:98`, `2026-06-16-the-imitation-game-state-of-policy-distillation-in-language-analysis.md:249`); source references l312-320 and l340-348.

- **name:** Privileged Context Self-Distillation
- **problem solved:** Long-running agents often need design-time information such as full logs, complete documents, or troubleshooting playbooks that cannot fit in the runtime context window.
- **inputs:**
  - Same base agent or model in two views: runtime view without privileged information and teacher view with privileged information.
  - Privileged information such as full session logs, stack traces, reference solutions, architecture diagrams, or complete source documents.
  - Paired task prompts where the teacher view can explain, correct, or compress better than the runtime view.
  - Difference signal between the two views, such as log-ratio, critique delta, or structured before/after correction.
  - Target surface such as prompt instructions, skills, memory writeback, eval examples, or fine-tuning data.
- **outputs:**
  - Runtime-usable behavior distilled from full-context or oracle-context runs.
  - Compact rules, skills, eval cases, or summaries that reduce dependence on loading the privileged source every time.
  - Evidence of which privileged facts changed the agent's action and which did not.
- **benefits:**
  - Converts expensive full-context analysis into reusable operating knowledge.
  - Supports context-management systems where the training or review phase sees more than production can carry.
  - Does not require a larger teacher model when the same model can be compared under different context views.
- **limitations:**
  - Privileged information can leak into outputs as false confidence or unreachable assumptions.
  - Secrets or private data must not be distilled into reusable artifacts.
  - Calibration must be measured under the runtime view, not under the privileged teacher view.

## 4. Consensus-Gated Privileged Information

Provenance: Phase 1 sections 1.4 and 6 (`2026-06-16-the-imitation-game-state-of-policy-distillation-in-language-analysis.md:72`, `2026-06-16-the-imitation-game-state-of-policy-distillation-in-language-analysis.md:243`); source references l415 and l464.

- **name:** Consensus-Gated Privileged Information
- **problem solved:** Retrieved documents, sub-agent findings, and teacher traces can be wrong or stale, so treating every privileged signal as equally trustworthy turns partial information into durable agent errors.
- **inputs:**
  - Candidate privileged information such as retrieved docs, logs, generated plans, reference answers, or sub-agent outputs.
  - Multiple tutor, evaluator, or sub-agent rollouts over the same candidate information.
  - Agreement metric such as exact answer match, rubric agreement, semantic consensus, or conflict count.
  - Trust threshold that decides whether the information can influence the next update or action.
  - Audit record of dissenting outputs and accepted consensus.
- **outputs:**
  - Accepted privileged information when independent rollouts agree above threshold.
  - Rejected, deferred, or escalated information when consensus is weak.
  - Evidence bundle that records why the signal was trusted.
- **benefits:**
  - Filters noisy retrieved context before it is promoted into prompts, skills, plans, or memory.
  - Maps naturally to multi-agent review councils where agreement is a gate, not a decoration.
  - Makes trust in privileged information explicit and inspectable.
- **limitations:**
  - Adds latency and token cost through repeated tutor or evaluator calls.
  - Correlated agents can agree on the same wrong answer.
  - High-value minority dissent may be lost if the consensus rule is too strict.

## 5. Asymmetric Failure Correction Router

Provenance: Phase 1 section 2.3 (`2026-06-16-the-imitation-game-state-of-policy-distillation-in-language-analysis.md:110`); source references l255-274.

- **name:** Asymmetric Failure Correction Router
- **problem solved:** Positive examples and negative failures have different mechanics, but many agent improvement loops route both through the same feedback path and let noisy failures dominate or stagnate.
- **inputs:**
  - Agent trace with decisions, tool calls, errors, user-visible outcome, and evaluator notes.
  - Success or failure signal at the step and trajectory levels.
  - Negative-feedback criteria such as verifier rejection, constraint failure, repeated retry, or confidence drop.
  - Stable correction channel for failures, such as targeted repair prompts, top-K alternatives, regression creation, or human review.
  - Reinforcement channel for successes, such as exemplars, demonstrations, or promoted prompt rules.
- **outputs:**
  - Failures routed to repair, regression, root-cause classification, or escalation.
  - Successes routed to examples, confidence-building, or reduced supervision.
  - Separate learning logs for what to imitate and what to correct.
- **benefits:**
  - Prevents failure feedback from being treated as a mirror image of success reinforcement.
  - Gives severe or high-variance failures a stabilizing path instead of repeatedly asking the same agent to self-correct.
  - Produces cleaner eval and backlog artifacts because every failure has a correction route.
- **limitations:**
  - Depends on reliable failure classification; false positives can over-route normal exploration into heavy review.
  - The correction channel must be tested, or it becomes another unverified fallback.
  - Low-risk tasks may not justify separate success and failure machinery.

## 6. Magnitude-Direction Verifier Split

Provenance: Phase 1 sections 2.5 and 6 (`2026-06-16-the-imitation-game-state-of-policy-distillation-in-language-analysis.md:126`, `2026-06-16-the-imitation-game-state-of-policy-distillation-in-language-analysis.md:241`); source references l435-464.

- **name:** Magnitude-Direction Verifier Split
- **problem solved:** An agent's internal confidence can reveal which decisions matter, but it cannot by itself prove whether the correction should push toward or away from that behavior.
- **inputs:**
  - Internal magnitude signal such as self-distillation delta, confidence gap, log-ratio, attention hotspot, or disagreement intensity.
  - External direction signal such as deterministic tests, verifier result, evaluator rubric, user approval, or human review.
  - Trace segment or output span to update.
  - Policy for combining confidence magnitude with verified direction.
  - Escalation rule when magnitude is high but direction is uncertain.
- **outputs:**
  - Weighted correction plan that spends effort where the agent signal is strong and the verifier has grounded the direction.
  - Approved, rejected, or escalated updates with separated rationale for magnitude and direction.
  - Audit trail showing whether a change was driven by model confidence, verifier result, or both.
- **benefits:**
  - Formalizes "trust but verify" for agent improvement loops.
  - Preserves dense model signal without letting the model be the sole judge of correctness.
  - Reduces information-leakage and overconfidence risks from pure self-distillation.
- **limitations:**
  - The verifier becomes a bottleneck; bad tests or rubrics can point confidently in the wrong direction.
  - Extra verification adds latency and operational cost.
  - Conflicting magnitude and direction signals need explicit tie-breaking or human escalation.

## 7. Adaptive Style Compression Teacher

Provenance: Phase 1 sections 2.4, 3.6, and 5.6 (`2026-06-16-the-imitation-game-state-of-policy-distillation-in-language-analysis.md:118`, `2026-06-16-the-imitation-game-state-of-policy-distillation-in-language-analysis.md:170`, `2026-06-16-the-imitation-game-state-of-policy-distillation-in-language-analysis.md:225`); source references l417-433 and l561.

- **name:** Adaptive Style Compression Teacher
- **problem solved:** Fixed summarization or token-budget heuristics either compress easy tasks too little, compress hard tasks too much, or remove uncertainty markers that downstream agents need.
- **inputs:**
  - Full trace, context, or reasoning artifact to compress.
  - Style instruction such as "be concise" or a domain-specific brevity rule.
  - Task difficulty signals such as verifier uncertainty, number of unresolved constraints, failed attempts, or dependency depth.
  - Candidate compressed output with explicit uncertainty, hedging, and unresolved-decision markers preserved.
  - Eval cases that compare downstream task success using compressed versus full context.
- **outputs:**
  - Compression behavior that is terse on easy tasks and preserves deliberation on hard tasks.
  - Summary or response format that keeps uncertainty markers instead of flattening them into false confidence.
  - Compression policy examples suitable for prompt, skill, or context-builder updates.
- **benefits:**
  - Saves tokens without reducing context management to a fixed truncation rule.
  - Aligns summary length with task difficulty rather than with a static budget alone.
  - Protects downstream agents from epistemic suppression caused by overconfident summaries.
- **limitations:**
  - Style compression can still suppress important hedging if not explicitly evaluated.
  - A single style instruction may conflict with domain requirements for auditability or legal precision.
  - Requires downstream evals; shorter traces are not automatically better traces.

## 8. Trajectory Failure Pattern Classifier

Provenance: Phase 1 sections 3, 5, and 6 (`2026-06-16-the-imitation-game-state-of-policy-distillation-in-language-analysis.md:136`, `2026-06-16-the-imitation-game-state-of-policy-distillation-in-language-analysis.md:193`, `2026-06-16-the-imitation-game-state-of-policy-distillation-in-language-analysis.md:245`); source references l483-495, l497-509, l511-521, l535-545, l547-559, and l561.

- **name:** Trajectory Failure Pattern Classifier
- **problem solved:** Long-running agent failures are often labeled generically as "bad output" or "tool failure", hiding whether the root mechanism was proxy metric drift, early-prefix drift, supervision expiry, diversity collapse, PI leakage, calibration gap, or epistemic suppression.
- **inputs:**
  - Complete trajectory trace with step-level events, tool calls, context snapshots, compressed summaries, and final outcome.
  - Step-level scores, verifier confidence, evaluator notes, and final task-level success metric.
  - Evidence of privileged-information use, summary compression, retry loops, and alternative sampled strategies.
  - Failure taxonomy derived from Phase 1 mechanisms.
  - Routing table from failure class to mitigation.
- **outputs:**
  - Failure classification attached to the trace.
  - Mitigation route such as trajectory-level eval, early re-anchoring, adaptive supervision stop, diversity restoration, verifier grounding, calibration check, or uncertainty-preserving summary.
  - Regression case or backlog item with the mechanism named, not just the symptom.
- **benefits:**
  - Prevents teams from optimizing step-level proxies when the real failure is trajectory-level.
  - Gives debugging and eval work a shared vocabulary grounded in failure mechanics.
  - Routes different failure modes to different fixes instead of adding generic retries.
- **limitations:**
  - Requires rich traces; without context snapshots and evaluator confidence over time, many classes collapse into "unknown".
  - Boundaries between classes can be subjective and need reviewer calibration.
  - This is a learning-loop classifier, not a substitute for ordinary code or infrastructure debugging.
